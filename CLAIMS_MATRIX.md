# CLAIMS_MATRIX.md

| Claim | Current evidence | Risk level | Missing proof | Suggested verification |
|---|---|---|---|---|
| PostgreSQL wire protocol support | `PgWireServer` + `PgSession` implement full startup negotiation, simple query, and extended query protocol (Parse/Bind/Describe/Execute/Sync/Close). `test_pgwire.cpp` covers 7 cases: connect, CREATE/INSERT, SELECT, transaction block, rollback, UPDATE, error handling. | **Low** | No SSL/TLS; auth is `AuthOk` with no actual credential check; no test for pipelining or cancellation. | Acceptable for a research DB; document no-auth in README. |
| SQL subset / parser / executor | `Parser` is a real recursive-descent parser covering CREATE TABLE, INSERT, SELECT, UPDATE (with arithmetic assignments), DELETE, BEGIN, COMMIT, ROLLBACK. WHERE supports multi-condition AND only. `parser.h` confirms token types and grammar rules. | **Low-Medium** | No JOINs, no aggregates (COUNT/SUM/etc.), no ORDER BY/LIMIT/GROUP BY, no subqueries, no OR predicates, no RETURNING clause. | Acceptable if documented. README/resume should state "limited SQL subset." |
| MVCC storage over RocksDB | `mvcc_store.h` directly includes `<rocksdb/db.h>` and wraps `unique_ptr<rocksdb::DB>`. `Get(snapshot_ts)` and `Put(write_ts)` implement versioned reads/writes. `MVCCIterator` filters by snapshot timestamp. `ReplayPut` for WAL recovery. | **Verified** | No evidence of column families or compaction tuning. | No action needed; core claim is solid. |
| Strict 2PL + serializable isolation | `LockManager` has Acquire/Release/ReleaseAll with FIFO wait queue. `Transaction` tracks read_set + write_set + locked_keys. `TxnManager::Prepare` calls `ValidateReadSet`. Tests: `SnapshotIsolation`, `ReadSetValidationDetectsConflict`, `WriteConflictAborts`, `LockManager` unit tests. | **Medium** | No test for write-skew, phantom reads, or named serializable anomalies. Read-set validation is OCC-style layered on strict 2PL writes; combined correctness under all scenarios is not test-proven. | Add write-skew and phantom read tests. Weaken claim to "serializability via strict 2PL + read-set validation; not formally verified." |
| WAL / durability | `WAL` has Append/Sync/AppendSync/Replay. `TxnManager::Recover()` replays WAL and re-applies committed data via `ReplayPut`. Tests: `WAL.AppendAndReplay`, `WAL.SyncDurability`, `WAL.TruncatedRecordIgnored`, `TxnManager.RecoverCommittedTxn/Aborted/Prepared`. `FaultTest.CommittedDataSurvivesRestart` and `FaultTest.AbortedDataNotVisibleAfterRestart` test at integration level. | **Medium** | Verified from `src/txn/wal.cpp`: `WAL::Sync()` calls `std::ofstream::flush()` only, with no explicit OS-level `fsync` / `FlushFileBuffers`. Power-loss durability remains TODO/VERIFY. | Add platform-specific durable sync and a crash/power-loss-oriented durability test. |
| Raft-style replication | `RaftLog` now supports path-backed persistence (`raft_log`) with load/append/truncate-rewrite behavior, and `RaftNode` now persists `current_term_`/`voted_for_` in `raft_meta` with load on startup. Added tests in `test/test_raft.cpp`: `RaftPersistence.LogSurvivesRestart`, `RaftPersistence.MetaSurvivesRestart`, `RaftPersistence.VotedForSurvivesRestart`. Existing election/replication tests remain in place. | **Medium** | TODO/VERIFY: tests not executed in this environment because configure/build is blocked by missing `RocksDB` package resolution after path flattening. Persistence writes use flush + rename but no explicit OS-level durability call. | Re-enable build, run targeted raft tests, then harden log/meta writes with explicit OS-level durability calls. |
| Cross-shard 2PC | `Coordinator::TwoPhaseCommit` implements parallel Prepare fan-out, all-vote-commit check, then parallel Commit (or parallel Abort on failure). `SingleShardCommit` optimizes 1-shard path. `FaultTest.MultiShardCommitAtomicity` tests multi-shard atomicity. Scan fans all shards in parallel and merges. | **Medium** | No coordinator durability: if coordinator crashes between Prepare and Commit, transaction outcome is lost. `FaultTest.SingleReplicaFailureNoDataLoss` is `GTEST_SKIP()`ped. | Document in README: "coordinator crash mid-2PC requires manual recovery or timeout abort - coordinator state is not persisted." |
| TPC-C benchmark harness | `bench/benchmark.cpp` implements `RunNewOrder` and `RunPayment` with warehouse/district/customer key model, latency tracking via mutex-protected vector, and warmup/duration config. | **Medium** | Only 2 of 5 TPC-C transaction types implemented (missing OrderStatus, Delivery, StockLevel). This is a TPC-C-inspired subset, not a conformant TPC-C run. Benchmark results should not be compared directly to industry TPC-C numbers. | Rename/describe as "TPC-C-inspired micro-benchmark (NewOrder + Payment)." |
| Fault/integration tests | 7 test files found: `test_mvcc_store`, `test_txn`, `test_raft`, `test_e2e`, `test_parser`, `test_pgwire`, `test_fault`. `FaultTest.SingleReplicaFailureNoDataLoss` is explicitly skipped. | **Medium** | CLAUDE.md claims 75 tests; count not independently verified. One skipped fault test reduces fault-tolerance evidence. | Run `ctest --test-dir build -C Release --output-on-failure` to confirm count and pass rate. |
